# 프로젝트 개요

## 문제 정의

커뮤니티 또는 중고거래 서비스에는 스팸, 사기, 어뷰징, 정책 위반과 관련된 사용자 신고가 지속적으로 유입된다.
운영팀은 한정된 리소스로 신고를 빠르게 검토해야 하며, 모든 신고를 동일한 우선순위로 다루면 대응 속도와 품질이 모두 떨어진다.

이 프로젝트의 목표는 신고를 접수한 뒤 AI 또는 규칙 기반 로직으로 1차 triage를 수행하고, 운영 큐로 자동 라우팅하는 최소 운영 자동화 시스템을 만드는 것이다.

## 핵심 목표

- 신고 입력을 안정적으로 저장한다.
- 신고 내용을 기반으로 `category`, `priority`, `requires_review`를 산출한다.
- 운영 큐(`fraud-review`, `spam-review`, `abuse-review`, `general-review`)로 자동 배정한다.
- 처리 흐름을 재시도 가능한 비동기 workflow로 분리한다.
- 분류 결과를 이벤트로 발행해 후속 운영 시스템이 구독할 수 있게 한다.

## 왜 이 주제인가

- 운영 자동화, 신뢰·안전, 정책 위반 탐지라는 당근 운영개발팀의 문제 영역과 직접 맞닿아 있다.
- FastAPI, Temporal, NATS JetStream의 역할 분담을 설명하기 좋다.
- 1~2일 안에 구현 가능한 범위로 잘라낼 수 있다.
- "정확한 모델"보다 "운영 가능한 구조"를 보여주기에 적합하다.

## 시스템 구성

### FastAPI

- 외부 진입점
- 신고 생성 및 조회 API 제공
- workflow 시작/재처리 요청 담당

### Temporal

- 신고 triage의 장기 실행 흐름 담당
- 단계별 activity 호출
- 실패 재시도 및 fallback 제어

### PostgreSQL

- 신고 원본
- 분류 결과
- 운영 큐 아이템 저장

### NATS JetStream

- 분류 완료 이벤트 전달
- 큐 라우팅 완료 이벤트 전달
- 후속 소비자 확장을 위한 비동기 경계 제공

## 도메인 언어

- `report`: 사용자가 접수한 신고 원본
- `classification`: 시스템이 내린 1차 판단 결과
- `queue item`: 운영 큐에서 검토할 대상
- `requires_review`: 사람 검토가 필요한지 여부
- `priority`: 운영 우선순위
- `category`: 정책/운영 분류 결과

## 성공 기준

- 신고 1건이 접수되면 비동기 triage가 정상 수행된다.
- 결과가 DB에 저장되고 API로 조회 가능하다.
- 운영 큐별로 신고 목록을 확인할 수 있다.
- workflow 실패 시 재시도 또는 fallback 상태가 남는다.
- 이벤트가 JetStream에 발행된다.
